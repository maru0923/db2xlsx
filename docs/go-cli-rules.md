# Go CLI ツール開発ルール

このドキュメントは、Go で安定した均質なコマンドラインツールを作るための標準ルールです。個別の事情がない限り、ここに書いた構成・ライブラリ・実装方針を採用します。

## 基本方針

- CLI は「人間が端末で使う UI」であり、出力・終了コード・ヘルプ・エラー文言まで API として扱う。
- Go の標準機能を優先し、外部ライブラリは CLI の一貫性、保守性、テスト容易性を明確に上げる場合に限って使う。
- コマンドの責務とビジネスロジックを分離し、CLI 層は入出力、フラグ解析、終了コードへの変換だけを担当する。
- グローバル状態、暗黙の環境依存、実行タイミングで変わる副作用を最小化する。
- すべてのサブコマンドは、ヘルプを読めば安全に実行できる粒度で設計する。

## 推奨ディレクトリ構成

```text
.
├── cmd/
│   └── toolname/
│       └── main.go
├── internal/
│   ├── cli/
│   │   ├── root.go
│   │   ├── <command>.go
│   │   └── exit.go
│   ├── config/
│   │   └── config.go
│   ├── app/
│   │   └── service.go
│   └── version/
│       └── version.go
├── pkg/
│   └── ...
├── testdata/
├── docs/
├── go.mod
├── go.sum
├── Makefile
├── README.md
└── .golangci.yml
```

### 配置ルール

- `cmd/<toolname>/main.go` は可能な限り薄くし、`os.Args`、`os.Stdout`、`os.Stderr`、終了処理の接続だけを行う。
- `internal/cli` に Cobra などのコマンド定義、フラグ定義、ヘルプ、終了コード変換を置く。
- `internal/app` や `internal/<domain>` に実処理を置き、CLI フレームワークに依存させない。
- `internal/config` に設定読み込み、環境変数、デフォルト値、バリデーションを集約する。
- `internal/version` にビルド時注入される `version`、`commit`、`date` を置く。
- 外部利用を本当に想定する再利用可能 API だけを `pkg/` に置く。迷う場合は `internal/` に置く。
- テスト用固定データは `testdata/` に置き、実行時データや生成物と混ぜない。

## 使うべきライブラリ

### 標準採用

- CLI フレームワーク: `github.com/spf13/cobra`
  - サブコマンド、POSIX 互換フラグ、ヘルプ、自動補完を標準化する。
  - 小規模で単一コマンドのみのツールでも、将来の拡張が見込まれる場合は Cobra を使う。
- ロギング: `log/slog`
  - 通常の人間向け出力とログを分離する。
  - `--log-format text|json` と `--log-level` を共通オプションにする。
- コンテキスト制御: `context`
  - すべての長時間処理、I/O、外部コマンド、ネットワーク処理に `context.Context` を渡す。
- テスト: 標準 `testing`、必要に応じて `github.com/stretchr/testify`
  - まず標準 `testing` を使い、複雑な検証やモックの見通しが良くなる場合だけ `testify` を使う。
- 静的解析: `golangci-lint`
  - CI で `go test ./...` と合わせて必ず実行する。

### 条件付き採用

- 設定ファイル: `github.com/spf13/viper`
  - 複数形式の設定ファイル、環境変数、フラグの統合が必要な場合だけ使う。
  - 単純な YAML/JSON 読み込みだけなら標準ライブラリや小さな専用パーサを優先する。
- TUI: `github.com/charmbracelet/bubbletea`
  - 進捗表示を超える対話的 UI が必要な場合だけ使う。
  - 通常の CLI と TUI モードは明確に分ける。
- 外部プロセス実行: `os/exec`
  - 必ず `exec.CommandContext` を使い、引数は文字列結合せず配列で渡す。
- テーブル表示: `github.com/olekukonko/tablewriter` など
  - 機械可読出力と人間向け表示を分離できる場合だけ使う。

### 原則避けるライブラリ

- メンテナンスが止まっている CLI フレームワーク。
- グローバル変数中心でテストしにくい設定・DI ライブラリ。
- 標準ライブラリで十分な処理を置き換えるだけの薄いラッパー。
- 出力の色付けや装飾だけを目的にした重い依存。

## コマンド設計

- コマンド名は動詞または明確な対象名にする。
  - 良い例: `tool scan`, `tool export`, `tool user list`
  - 悪い例: `tool do`, `tool run2`, `tool misc`
- サブコマンドは `APP VERB NOUN --FLAG` または `APP NOUN VERB --FLAG` のどちらかに統一する。
- 破壊的操作には `--dry-run` を用意する。
- 上書き、削除、送信など戻せない操作には `--yes` または `--force` を要求する。
- すべてのコマンドに短い `Short` と実用的な `Long`、必要に応じて `Example` を書く。
- フラグ名は kebab-case にする。
  - 良い例: `--output-format`, `--config-file`
  - 悪い例: `--outputFormat`, `--config_file`
- 短縮フラグは本当に頻出するものだけにする。
  - 推奨: `-o` for `--output`, `-f` for `--file`, `-v` for `--verbose`
  - 禁止: 意味が推測できない短縮、サブコマンド間で意味が変わる短縮。
- デフォルト値は安全側に倒す。削除、送信、課金、公開をデフォルトにしない。

## 入出力ルール

- 通常結果は `stdout`、エラー・警告・ログ・進捗は `stderr` に出す。
- 機械可読出力を持つコマンドは `--output text|json|yaml` を共通化する。
- `json` 出力は安定したスキーマとして扱い、安易にフィールド名や型を変えない。
- 人間向け出力では色に依存して意味を伝えない。
- 色出力は `--color auto|always|never` を共通化する。
- 進捗表示は TTY のときだけ出す。パイプやリダイレクト時は抑制する。
- 成功時に不要な文言を出さない。スクリプト利用を妨げる雑なログを `stdout` に混ぜない。
- 標準入力を読むコマンドは、ヘルプとエラーでその仕様を明示する。

## エラーと終了コード

- エラーは握りつぶさず、必ず呼び出し元に返す。
- `panic` はプログラミングエラーに限定し、ユーザー入力や外部要因では使わない。
- エラー文言はユーザーが次に何をすればよいか分かる形にする。
- 内部エラーの詳細、パス、API レスポンス、SQL、トークンなどを不用意に表示しない。
- 終了コードはツール全体で統一する。

```text
0  success
1  general error
2  invalid usage or validation error
3  not found
4  conflict or already exists
5  external dependency error
6  timeout or canceled
7  permission denied
```

- `context.Canceled` と `context.DeadlineExceeded` は区別して扱う。
- 複数エラーを返す場合は `errors.Join` を使い、表示側で読みやすく整形する。
- エラー比較は文字列比較ではなく `errors.Is` / `errors.As` を使う。

## 設定と優先順位

設定値の優先順位は必ず次の順にする。

```text
CLI flag > environment variable > config file > default value
```

- 環境変数は `<TOOLNAME>_<NAME>` の形式にする。
  - 例: `ACMECTL_API_TOKEN`, `ACMECTL_LOG_LEVEL`
- 設定ファイルの探索場所は明示し、暗黙に多数の場所を探しすぎない。
- `--config` で指定された設定ファイルが存在しない場合はエラーにする。
- 自動検出の設定ファイルが存在しない場合はデフォルト値で動作する。
- 秘密情報は設定ファイルに平文保存する設計を避け、環境変数または OS の secret store を優先する。
- 設定のバリデーションは実処理の前にまとめて実行する。

## ロギング

- 通常出力とログを混ぜない。
- `slog` を標準ロガーとして使い、構造化ログを前提にする。
- 共通フラグとして `--log-level debug|info|warn|error` を用意する。
- `--verbose` は人間向けの詳細表示、`--log-level` はログ制御として扱い、意味を混同しない。
- トークン、パスワード、個人情報、署名付き URL はログに出さない。
- リトライ、外部 API 呼び出し、ファイル更新などの重要な副作用は `debug` または `info` で追えるようにする。

## 実装ルール

- `main` から直接ビジネスロジックを呼ばない。必ず `cli.Execute(ctx, in, out, errOut)` のような入口を作る。
- 標準入出力は直接参照せず、`io.Reader` / `io.Writer` として注入できるようにする。
- ファイルシステム操作は小さな境界に集約し、テストで差し替えられるようにする。
- パス操作には `path/filepath` を使い、OS 固有のパス区切り文字を文字列結合で扱わない。
- 時刻は `time.Now()` をロジック内に散らさず、必要に応じて clock を注入する。
- ランダム値、UUID、現在時刻、環境変数はテストで固定できるようにする。
- 外部 API や外部コマンド呼び出しにはタイムアウトを設定する。
- リトライは対象エラー、回数、待機時間、総タイムアウトを明示する。
- 並行処理は `errgroup` などでキャンセル伝播とエラー回収を明確にする。
- 共有変数への並行アクセスは mutex、channel、atomic のいずれかで保護する。
- `init()` に実処理を書かない。登録処理も最小限にする。
- `os.Exit` は `main` 以外で呼ばない。

## クロスプラットフォーム対応

- 特別な理由がない限り、CLI は Windows / Linux / macOS で動作する前提で実装する。
- パス、改行コード、ファイル名、ファイル権限、環境変数、外部プロセス呼び出しの OS 差異を設計時に確認する。
- 入力ファイルの解析は CRLF / LF のどちらにも対応し、改行コードに依存したパースを書かない。
- 出力ファイル名には Windows で使用できない文字を含めない。
  - 禁止文字の例: `<`, `>`, `:`, `"`, `/`, `\`, `|`, `?`, `*`
- `:` を含む時刻表現など、特定 OS で無効なファイル名になりやすい形式を避ける。
  - 良い例: `YYYYMMDD_HHMMSS.log`
  - 悪い例: `YYYY-MM-DD HH:MM:SS.log`
- 環境変数の参照形式は OS ごとのシェル構文に依存させず、ツール内の設定形式では `${NAME}` などの共通表現を定義する。
- ファイル権限は Unix と Windows で意味が異なるため、権限値だけに依存したセキュリティ設計にしない。
- 外部コマンドを呼ぶ必要がある場合は、Windows / Linux の実行ファイル名、引数、終了コード、文字コード差異を確認する。
- ネイティブライブラリや OS 固有ドライバに依存するライブラリを採用する場合は、README に OS ごとの追加セットアップを明記する。
- クロスコンパイルが必要なツールでは、`GOOS` / `GOARCH` ごとのビルド手順を README またはリリース手順に書く。
- Windows と Linux の両方で同じフラグ体系、終了コード、標準出力/標準エラーの扱いを維持する。

## テストルール

- CLI 層のテストでは `stdout`、`stderr`、終了コード、エラー文言を検証する。
- ビジネスロジックは CLI フレームワークに依存しない単体テストを書く。
- ファイル入出力を伴うテストは `t.TempDir()` を使う。
- 環境変数を使うテストは `t.Setenv()` を使う。
- パス処理、改行コード、ファイル名生成、環境変数展開は OS 差異を意識した単体テストを書く。
- ゴールデンファイルは `testdata/` に置き、更新方法を README またはテストコメントに書く。
- 並行処理、キャンセル、タイムアウト、リトライは必ずテストする。
- バグ修正には再発防止テストを追加する。
- CI では最低限次を実行する。

```sh
go test ./...
go vet ./...
golangci-lint run
```

## ビルドとリリース

- バイナリ名はリポジトリ名またはプロダクト名と一致させる。
- バージョン情報は `tool version` で確認できるようにする。
- `version`、`commit`、`date` は `-ldflags` で注入する。
- リリースビルドは OS / ARCH ごとに生成し、チェックサムを公開する。
- Windows 向けバイナリには `.exe` 拡張子を付ける。
- README には Windows と Linux の実行例・ビルド例をそれぞれ記載する。
- `go mod tidy` 済みの状態を保つ。
- 依存追加時はライセンス、メンテナンス状況、推移的依存を確認する。
- 補完スクリプトを提供する。
  - `tool completion bash`
  - `tool completion zsh`
  - `tool completion fish`
  - `tool completion powershell`

## セキュリティ

- シェルコマンドを文字列結合で組み立てない。
- ユーザー入力をパス、URL、SQL、テンプレート、正規表現に使う場合は必ず検証する。
- パストラバーサルを防ぐため、ユーザー指定パスは `filepath.Clean` と許可ディレクトリ確認を行う。
- 一時ファイルは `os.CreateTemp` / `os.MkdirTemp` を使う。
- ファイル権限は最小限にする。秘密情報を含むファイルは `0600` を基本にする。
- TLS 検証を無効化しない。
- API トークンや秘密情報を CLI 引数に渡す設計を避ける。履歴に残るため、環境変数か標準入力を使う。
- エラーメッセージとログに秘密情報を含めない。

## 互換性

- 公開済みコマンド名、フラグ名、JSON 出力スキーマは後方互換を保つ。
- 変更が必要な場合は非推奨期間を設け、ヘルプと警告に削除予定バージョンを書く。
- 機械可読出力に人間向けメッセージを混ぜない。
- デフォルト動作の変更は破壊的変更として扱う。
- エラー文言はある程度変わり得るが、終了コードと JSON エラー形式は安定させる。

## 実施してはいけないこと

- `main` や Cobra の `Run` に業務ロジックを直接書く。
- `fmt.Println` でどこからでも出力する。
- `stdout` にログ、警告、進捗を出す。
- ユーザー入力エラーで `panic` する。
- `os.Exit` を `main` 以外から呼ぶ。
- テスト不能なグローバル変数に設定、ロガー、HTTP クライアント、時刻を閉じ込める。
- 環境変数をロジックの奥で直接読む。
- シェルコマンドを `sh -c` / `cmd /c` で安易に実行する。
- タイムアウトなしでネットワークや外部コマンドを実行する。
- `context.Background()` を処理の途中で新しく作り、キャンセル伝播を切る。
- エラーを `_` で捨てる。
- エラー文字列で分岐する。
- 秘密情報をログ、エラー、JSON 出力、デバッグ出力に含める。
- 破壊的処理を確認なし・dry-run なしで実行する。
- 設定値の優先順位をコマンドごとに変える。
- ヘルプのない隠しフラグを常用する。
- マジックナンバーの終了コードを各所に散らす。
- テストで実ユーザーのホームディレクトリ、実ネットワーク、本番 API に依存する。
- 依存ライブラリを用途、ライセンス、保守状況を確認せず追加する。

## レビュー観点

- コマンド名、フラグ名、ヘルプ、例は一貫しているか。
- `stdout` / `stderr` / ログの分離ができているか。
- 終了コードがルール通りか。
- 破壊的操作に `--dry-run` と確認手段があるか。
- 設定の優先順位が守られているか。
- `context`、タイムアウト、キャンセルが伝播しているか。
- Windows / Linux のパス、改行コード、ファイル名、環境変数の差異に対応できているか。
- 秘密情報が出力されないか。
- CLI 層とビジネスロジックが分離されているか。
- テストが入出力、終了コード、エラー、設定、キャンセルをカバーしているか。
- 新規依存の必要性と保守性が説明できるか。

## 新規コマンド追加時のチェックリスト

- [ ] コマンド名とフラグ名が命名規則に合っている。
- [ ] `Short`、`Long`、`Example` が書かれている。
- [ ] `stdout` と `stderr` の使い分けが正しい。
- [ ] `--output` 対応が必要か判断済み。
- [ ] `--dry-run`、`--yes`、`--force` が必要か判断済み。
- [ ] 設定値の優先順位が守られている。
- [ ] Windows / Linux のパス、改行コード、ファイル名、環境変数の差異を考慮している。
- [ ] タイムアウトとキャンセルに対応している。
- [ ] 終了コードが定義済みのものに収まっている。
- [ ] 単体テストと CLI テストがある。
- [ ] README または docs に使い方を追加した。
